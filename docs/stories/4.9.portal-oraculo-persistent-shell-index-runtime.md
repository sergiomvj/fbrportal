# Story 4.9: Portal UI / Oraculo Persistent Shell, History, Index, and Cache Hardening

## Status

Draft

## Story

**As a** portal user,  
**I want** Oraculo integrated into the portal shell with persisted history and reliable source indexing,  
**so that** contextual answers are available across modules without relying on process-local cache as source of truth.

## Problem to Fix

Portal/Oraculo is partially implemented, but system-resource review found local cache usage and remaining uncertainty around persistent history, pgvector/indexing, shell integration, and readiness.

Current evidence:

- `apps/portal/src/lib/oraculo/query.ts` uses `SOURCE_CACHE = new Map`.
- PRD Oraculo requires Supabase history, pgvector chunks, webhook/index worker, and shell integration.

## What Must Be Implemented

1. Revalidate shell integration:
   - main navigation entry;
   - panel open/closed state;
   - responsive behavior;
   - authenticated context;
   - error/empty/loading states.
2. Persist history:
   - user/company scoped history table;
   - query, response summary, sources, timestamps;
   - RLS policies;
   - route reads/writes history.
3. Implement or verify source index persistence:
   - chunks table;
   - pgvector or documented equivalent;
   - source metadata for file/module/story/PRD;
   - update timestamp and commit/source hash when available.
4. Rework local cache:
   - `SOURCE_CACHE` may remain as short-lived optimization only;
   - it must not be source of truth for user history or indexed corpus;
   - add TTL/invalidation if retained.
5. Implement indexing workflow/readiness:
   - documented worker or command to index repo sources;
   - readiness response when index is missing/stale;
   - tests for stale/missing index behavior.
6. Improve answer evidence:
   - return cited source paths;
   - avoid hallucinated module facts;
   - respect module boundaries and user access where applicable.

## What Must Be Changed or Reworked

- Replace any history-only-in-memory behavior with Supabase.
- Make cache behavior explicit and bounded.
- Update Oraculo story evidence if it currently marks shell done without persistent history/index verification.

## Acceptance Criteria

1. Oraculo is reachable from the portal shell.
2. Queries and responses are persisted per user/company.
3. Source index is persisted or readiness reports that indexing is missing.
4. Local `Map` cache is not source of truth.
5. Answers include source references.
6. Tests cover shell, history, index readiness, cache TTL/invalidation, and error states.

## Test Requirements

- Migration tests for Oraculo history/chunks if migrations are added.
- Route tests for query persistence and source references.
- Component tests for shell integration states.
- Cache tests proving miss/stale behavior falls back to persistent/index source.

- Oraculo is shell-integrated and persistent for history/index requirements.
- Cache is documented as optimization only.
- Gates pass and evidence is updated.
