# Story 4.3: FBR-Click Persistent CRM, Event Timeline, and Audit Runtime

## Status

Draft

## Story

**As a** Click operator,  
**I want** deals, stages, messages, tasks, agents, and cross-module events persisted with audit history,  
**so that** Click can act as the durable CRM hub for Leads, MKT, and Sales.

## Problem to Fix

Click route and UI behavior exists, but the system-resource review found the domain is still backed by fixtures and arrays.

Current evidence:

- `apps/portal/src/lib/click/store.ts` imports fixtures.
- Deals, messages, tasks, agents, and history are process-local arrays.
- Cross-module contracts are tested, but events are not proven durable or append-only.

## What Must Be Implemented

1. Create Click persistence migrations:
   - deals;
   - deal stage history;
   - messages;
   - tasks;
   - cross-module events;
   - agents;
   - audit log/export rows.
2. Enable and test RLS:
   - all user/company data scoped by company;
   - no cross-company deal visibility;
   - audit/event rows read-only except through controlled append paths.
3. Create persistent Click store:
   - recommended file: `apps/portal/src/lib/click/store-db.ts`;
   - preserve existing route contracts where valid;
   - use explicit mapping between API schema and table rows.
4. Rework manual deal creation:
   - insert deal row;
   - insert initial stage history row;
   - insert audit event;
   - reject invalid payload and missing context.
5. Rework `lead.qualified` ingestion:
   - dedupe by source event/deal identity;
   - preserve full lead source context;
   - create or update deal according to PRD rules;
   - append cross-module event.
6. Rework `strategy.exported` ingestion:
   - persist MKT export event;
   - attach campaign/strategy metadata to a deal or account record according to existing schema;
   - keep idempotency by event/export id.
7. Rework stage transitions:
   - validate official stage order/rules;
   - persist previous and next stage;
   - append actor, timestamp, reason, and correlation id.
8. Rework messages and tasks:
   - persist all messages/tasks;
   - preserve actor and timestamps;
   - prevent updates/deletes from mutating audit history.
9. Rework Click -> Sales event:
   - emit `deal.closed` from persisted deal state;
   - record delivery status, retry/failure, and response;
   - add readiness check for Sales endpoint config.

## What Must Be Changed or Reworked

- Runtime routes must stop using `apps/portal/src/lib/click/store.ts` as source of truth.
- Existing fixtures can remain only as tests/seeds.
- `docs/stories/3.2.1`, `3.2.2`, and `3.2.3` must be corrected if they currently say complete while runtime is not persistent.
- Audit export must query persisted audit/event tables.

## Acceptance Criteria

1. Manual deal creation persists deal, stage history, and audit event.
2. `lead.qualified` creates/updates a persisted deal and is idempotent.
3. `strategy.exported` persists a cross-module event and is idempotent.
4. Stage transitions are validated and persisted append-only.
5. Messages and tasks are persisted and company-scoped.
6. Audit export returns persisted audit data, not array history.
7. `deal.closed` is emitted from persisted state and delivery status is observable.
8. Tests prove no route imports the fixture-backed store for production behavior.

## Test Requirements

- Migration tests for Click tables, RLS policies, indexes, and audit append-only constraints.
- Store tests for manual deals, lead ingestion, MKT export ingestion, stage moves, messages, tasks, and audit.
- Route tests for all existing Click proxy routes using the persistent store.
- Contract tests:
  - Leads -> Click `lead.qualified`;
  - MKT -> Click `strategy.exported`;
  - Click -> Sales `deal.closed`.
- Regression guard that runtime routes do not import fixture-backed store.
