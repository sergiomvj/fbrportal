# Story 4.1: FBR-Leads Runtime Supabase Source Capture and Core Entities

## Status

In Progress - Implementing Persistent Store

## Story

**As a** Leads operator,  
**I want** source capture, leads, ICPs, domains, runs, and records persisted through Supabase with RLS,  
**so that** captured leads survive process restarts, respect company isolation, and can feed scoring, cadence, and Click handoff reliably.

## Problem to Fix

Story 3.1.1 created a functional source capture slice and a migration, but the operational route still imports the in-memory Leads store.

Current evidence:

- `supabase/migrations/20260514000001_create_leads_source_capture_tables.sql` creates the persistence foundation.
- `apps/portal/src/app/api/proxy/leads/source-runs/route.ts` now imports `captureLeadsFromSourceDb`, `getSourceRunDb`, and `listSourceRunsDb` from the persistent source-capture path.
- `apps/portal/src/lib/leads/source-capture.ts` now returns created leads alongside source records and guards persisted run IDs before updating run status/counters.
- `apps/portal/src/lib/leads/store-db.ts` now persists `normalized_lead_id` on source records.
- `apps/portal/src/lib/leads/store.ts` keeps `leads`, `icps`, `domains`, `sourceRuns`, and `sourceRecords` in arrays.
- `sourceRuns.push(...)` and `sourceRecords.push(...)` prove writes are process-local.

The source-runs runtime route is now on the persistent path, but broader Leads conformity still requires replacing the remaining in-memory core store surfaces and adding Supabase-backed store tests for the DB path.

## What Must Be Implemented

1. Create a persistent Leads runtime store, following the MKT pattern:
   - recommended file: `apps/portal/src/lib/leads/store-db.ts`;
   - use `getSupabaseClient` or the local project Supabase helper already used by MKT;
   - keep the existing `LeadsRequestContext` shape or introduce a compatible one;
   - map app fields to table columns explicitly;
   - return the existing API response contracts unless a PRD-backed contract correction is required.
2. Move source capture writes from arrays to Supabase:
   - create `leads_source_runs` row at run start;
   - insert one `leads_source_records` row per source payload;
   - persist normalized lead rows into `leads_leads`;
   - update run counters and terminal status after processing;
   - store `raw_payload`, `normalized_payload`, `source_key`, duplicate reason, and `fail_reason`.
3. Move core reads used by source capture to Supabase:
   - leads lookup for deduplication;
   - ICP lookup if needed by normalization;
   - domains lookup if source/cadence readiness is touched;
   - run listing and run detail with records.
4. Preserve and harden deduplication:
   - dedupe by email, CNPJ, website, and stable source key;
   - scope dedupe by `company_id`;
   - never mark another company lead as duplicate;
   - persist duplicate records for observability even when no new lead is created.
5. Enforce RLS-oriented context:
   - every write must include `company_id`;
   - every write must include `created_by`/`user_id` where table schema requires it;
   - every read must filter by `company_id`;
   - route tests must prove cross-company isolation at the store/query layer, even if Supabase RLS is mocked.
6. Refactor the source-runs route:
   - replace import from `@/lib/leads/store` with the persistent store;
   - avoid fallback to in-memory store in production code;
   - if local/demo seed data is still needed, keep it behind explicit test/dev helpers, not the route.
7. Update documentation and story evidence:
   - update `docs/stories/3.1.1.fbr-leads-data-foundation-source-capture.md`;
   - update `docs/stories/2.4.fbr-leads-conformity-alignment.md`;
   - link this story from `docs/backlog-pendente-por-modulo-prds.md` if that file remains the active backlog index.

## What Must Be Changed or Reworked

- Rework `apps/portal/src/app/api/proxy/leads/source-runs/route.ts` so it no longer depends on `@/lib/leads/store`.
- Split reusable normalization/dedup helpers out of the old store if needed, for example into `apps/portal/src/lib/leads/source-capture.ts`.
- Keep old in-memory helpers only for tests or remove them from runtime paths.
- Replace tests that only assert array mutation with tests that assert Supabase insert/select/update calls.
- Add a regression test that fails if `source-runs/route.ts` imports `@/lib/leads/store`.

## Acceptance Criteria

1. `POST /api/proxy/leads/source-runs` creates a persisted run and persisted source records.
2. `GET /api/proxy/leads/source-runs` reads persisted runs scoped to the request company.
3. `GET /api/proxy/leads/source-runs?id=...` returns a persisted run plus persisted records.
4. Deduplication is company-scoped and persists duplicate records with reason.
5. Failed source rows persist `fail_reason` and do not corrupt the run counters.
6. A process restart would not lose run/record/lead data because route state is no longer process-local.
7. Tests prove the route uses the persistent store and does not import the old in-memory store.
8. Evidence is registered in the 2.4, 3.1.1, and 4.1 stories.

## Test Requirements

- Unit tests for source normalization and stable source key generation.
- Store tests mocking Supabase:
  - insert run;
  - insert source records;
  - insert new leads;
  - detect duplicates;
  - update run status/counters;
  - filter by company.
- Route tests:
  - POST success for LinkedIn, CNPJ.biz, Google Maps, and site payloads;
  - POST partial failure;
  - GET list;
  - GET detail;
  - missing/invalid context rejection;
  - regression guard against importing `@/lib/leads/store`.
- Migration test:
  - tables required by this runtime exist;
  - required RLS policies exist;
  - required indexes/unique constraints for dedupe exist or are intentionally handled in code with documented reason.
