## Story

**As a** FBR super admin,
**I want** portal data migrations, user administration, health checks, user settings, and end-to-end security coverage,
**so that** the Portal UI can be operated safely after auth, shell, embedding, and notifications are implemented.

## Acceptance Criteria

1. Portal tables `profiles`, `user_roles`, `notifications`, and `empresas` exist with RLS, super admin policy, idempotent migrations, duplicate-role constraint, and dev seed data. [Source: `fbr-portal-docs/01_PORTAL_UI/TASKLIST.md#TASK-P18`]
2. `/admin/users` provides super-admin-only user listing, filters, create/edit modal, active toggle with `ConfirmDialog`, access log drawer, and protections against self-disable and last-super-admin removal. [Source: `fbr-portal-docs/01_PORTAL_UI/TASKLIST.md#TASK-P19`]
3. `/admin/health` provides super-admin-only health API/UI for all 9 modules, per-module 5s timeout, latency, degraded status >2s, and 30s polling cleanup. [Source: `fbr-portal-docs/01_PORTAL_UI/TASKLIST.md#TASK-P20`]
4. `/settings` supports company switching, session `empresaId` update, notification preferences persisted in `profiles.preferences`, safe defaults for null/corrupt preferences, and immediate preference reflection. [Source: `fbr-portal-docs/01_PORTAL_UI/TASKLIST.md#TASK-P21`]
5. E2E suite covers login, sidebar navigation, module switch, logout, route guard, realtime notification mock, admin access, and security review checklist. [Source: `fbr-portal-docs/01_PORTAL_UI/TASKLIST.md#TASK-P22`]
6. Security review verifies no raw Supabase UUID, token, or sensitive session value appears in console logs, URLs, response bodies, or localStorage.
7. Quality gates pass: lint, typecheck, tests, E2E/security checks where configured, and build if available.
