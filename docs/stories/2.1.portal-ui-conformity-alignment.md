## Story

**As a** FBR Portal platform owner,  
**I want** the portal shell, auth, embedding, notifications, admin/security, and shared navigation layers reviewed and corrected against the original docs,  
**so that** every module runs inside a portal foundation that matches the intended architecture, information flow, and operating concept.

## Acceptance Criteria

1. The implemented Portal UI scope is reviewed against `fbr-portal-docs/01_PORTAL_UI/PRD.md`, `fbr-portal-docs/01_PORTAL_UI/SPEC.md`, `fbr-portal-docs/01_PORTAL_UI/TASKLIST.md`, and `fbr-portal-docs/00_GERAL/DOCUMENTO_GERAL.md`, with every shipped capability marked as conformant, partial, or missing.
2. Authentication/session flow matches the original architecture: login through Next.js, `iron-session` cookie, proxy-only backend access, `X-User-Id` propagation, protected routes, and no token/session leakage to UI or localStorage.
3. Portal shell behavior matches the intended concept: sidebar, module routing, module headers, notifications, and global UX patterns are consistent across all module entry points.
4. Embedding and cross-module communication match the original contract: `AUTH_TOKEN`, `NAVIGATE`, `NOTIFICATION`, `MODULE_READY`, and `CROSS_MODULE_EVENT`, with the parent portal remaining the only cross-module navigation authority.
5. Admin/settings/security surfaces match the documented platform responsibilities, including environment validation, role-aware access, and global governance views where required by the original docs.
6. All identified Portal UI gaps produce concrete code changes, updated tests, and updated story/file lists without inventing new platform features outside the original docs.
7. Quality gates pass for the final conformity increment: `npm run lint`, `npm run typecheck`, `npm test`, and `npm run build`.
