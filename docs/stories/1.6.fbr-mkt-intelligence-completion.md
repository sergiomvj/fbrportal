## Story

**As a** marketing strategist,
**I want** FBR-MKT to process uploaded documents into diagnostics, strategies, copy, calendars, roadmap, chat, and exports,
**so that** marketing plans can be generated, reviewed, versioned, and sent to FBR-Click.

## Acceptance Criteria

1. Visual design implements MKT CSS variables, 8 KPI hero, 9 module cards, 7-pillar visualization, 6 agent cards, service loop, and onboarding flow.
2. Data model includes the 16 tables/entities listed in `TASKS_MKT.md`.
3. Components include strategy builder, diagnostic view, strategy master, copywriting generator, 90-day calendar, contextual chat, and PDF/PPTX export.
4. BullMQ job flow, SSE progress schema, worker orchestration, export generation, chat context, and version snapshot logic match the SPEC.
5. Arva integration supports 6 MKT slots: Extrator, Estrategista, Redator, Calendario, Exportador, Onboarding.
6. Integration sends exported strategy to FBR-Click via Portal Bridge.
7. Tests cover upload validation, worker stages, SSE progress, strategy JSON, copy/calendar generation, chat context, exports, and Click integration.

### Critical Issues

1. Executor assignment does not pass the validator in `.aiox-core/development/tasks/validate-next-story.md`: `quality_gate: "@qa"` is not in the workflow's allowed quality gate list (`@architect`, `@dev`, `@pm`). The story index also lists `@qa`, so the source artifacts need alignment before approval.
2. The story is not self-contained for implementation. Dev Notes point to `PRD_SPEC_TASKLIST.md`, `TASKS_MKT.md`, and `DOCUMENTO_GERAL.md`, but the workflow requires enough pulled context that the dev agent should not need to read external architecture/source documents to implement.
3. Scope is too broad for one implementation-ready story: it combines visual design, 16-entity data model, API routes, BullMQ workers, SSE, Arva integration, PDF/PPTX export, chat, versioning, FBR-Click integration, and all `TASK-MK01` through `TASK-MK20`.
4. Data model acceptance is inconsistent across sources: AC2 requires 16 tables/entities from `TASKS_MKT.md`, while the SPEC sprint 1 database section defines 4 core tables (`estrategias`, `diagnosticos`, `estrategia_versoes`, `chat_mkt`). The relationship between the 4-table sprint SPEC and 16-entity completion checklist must be made explicit.
5. Production paths are not aligned with the current repo structure. The story expects `apps/modules/mkt`, `backend/api/mkt`, BullMQ workers, export services, and storage integration; the current repository has `apps/portal`, shared `packages/*`, and no root `backend` directory.
