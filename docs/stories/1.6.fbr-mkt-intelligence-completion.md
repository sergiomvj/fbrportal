**As a** marketing strategist,
**I want** FBR-MKT to process uploaded documents into diagnostics, strategies, copy, calendars, roadmap, chat, and exports,

### Backend / Worker Contract From Source Docs

- API route shape follows `/api/proxy/{modulo}/{entidade}[/{id}[/{acao}]]`.
- Frontend must not call backend directly; requests go through Next.js proxy routes.
- Required MKT routes from source docs include create/list/detail strategy, status SSE, approve diagnostic, get strategy content, regenerate, copy, chat GET/POST, versions, calendar, roadmap, export, and send to FBR-Click.
- Worker orchestration from source docs is: Upload -> Extrator Bot -> Diagnostico -> Estrategista Bot -> Estrategia, then Redator Bot and Calendario Bot, then Exportador Bot.
- Chat context includes `estrategia_id`, `empresa_id`, `diagnostico`, active strategy version, generated copy, 90-day calendar, last-turn summary, niche, and preferred tone of voice.
- Chat limits from source docs: maximum 50 messages per chat session, last 20 turns sent to the model, automatic summary every 10 turns.
- Versioning from source docs: every generation/regeneration inserts a new immutable strategy version; existing records in `estrategia_versoes` are never altered.

### Security / Tenancy Requirements

- Every protected request must derive user context from the authenticated session/proxy path and enforce tenant isolation.
- Every table with user or organization data must have RLS or an equivalent approved isolation mechanism before UI access is allowed.
- Upload validation must use magic bytes and must not trust browser `Content-Type`.
- Sensitive configuration must not use `NEXT_PUBLIC_`.
- Rate limiting is required for sensitive routes; chat source guidance specifies 10 messages/minute per user per strategy.
- Agent actions and FBR-Click sends need immutable audit evidence through the approved log/table decision.
