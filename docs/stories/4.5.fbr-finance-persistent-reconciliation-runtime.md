# Story 4.5: FBR-Finance Persistent Reconciliation Worker and Audit Runtime

## Status

Draft

## Story

**As a** finance analyst,  
**I want** receivables, payables, reconciliation jobs, pendencies, audit, forecasts, reports, and alerts persisted,  
**so that** finance operations survive failures and satisfy the PRD audit model.

## Problem to Fix

Finance has migrations and functional tests, but runtime still keeps financial entities and reconciliation jobs in memory.

Current evidence:

- `apps/portal/src/lib/finance/store.ts` keeps receivables, payables, cost centers, thresholds, reconciliation items, and reconciliation jobs in arrays.
- PRD Finance requires async reconciliation, durable jobs, retry/failure state, human pendencies, audit append-only, reports, and anomaly alerts.

## What Must Be Implemented

1. Wire runtime to existing Finance migrations:
   - confirm table names and fields in `supabase/migrations/20260401000001_create_finance_tables.sql`;
   - create a persistent Finance store;
   - map existing route contracts to Supabase rows.
2. Persist receivables:
   - create/list/update receivables in DB;
   - receive Sales intake into persisted receivable;
   - append audit event on creation/update.
3. Persist payables and approvals:
   - create/list/update payables;
   - enforce approval thresholds;
   - append audit events for approval/rejection.
4. Persist cost centers and P&L inputs:
   - use DB as source of truth;
   - derive dashboard KPIs from persisted state.
5. Rebuild reconciliation runtime:
   - create persisted reconciliation job;
   - support pending, processing, done, failed;
   - store progress counters;
   - store error messages and retry metadata;
   - make status route read persisted job.
6. Rebuild matching and pendencies:
   - store imported bank/statement items;
   - match against persisted receivables/payables;
   - create persisted pendencies for unsafe matches;
   - implement approve/reject/create-entry against DB;
   - preserve full audit trail.
7. Implement worker strategy:
   - if BullMQ/Redis is not available in repo, implement a documented Supabase-backed job runner equivalent;
   - no process-local array may be the only job source of truth;
   - retries and timeouts must be represented in persisted state.
8. Rework reports/forecast/alerts:
   - generate from persisted financial data;
   - persist report/export jobs if asynchronous;
   - persist anomaly alerts and dedupe them.

## What Must Be Changed or Reworked

- Replace `reconciliationJobs` and `reconciliationItems` arrays as runtime source of truth.
- Update Story 3.7.2 if it currently says complete while the worker is in-memory.
- Rework tests to assert DB-store calls and persisted state transitions.
- Keep local fixtures only as deterministic seeds for tests.

## Acceptance Criteria

1. Sales intake creates persisted receivable and audit row.
2. Payables and approvals are persisted with audit.
3. Reconciliation run creates persisted job and persisted statement items.
4. Job status route reads persisted job state.
5. Unsafe matches create persisted human pendencies.
6. Approve/reject/create-entry updates persisted records and audit.
7. Reconciliation survives process restart by design because state is in DB/queue.
8. Forecast, report, and alert surfaces read persisted data.

## Test Requirements

- Migration tests for finance tables, RLS, audit triggers, and append-only constraints.
- Store tests for receivables, payables, approvals, cost centers.
- Reconciliation tests for:
  - job creation;
  - progress update;
  - failed job;
  - retry scheduling;
  - exact match;
  - divergence;
  - human pendency lifecycle;
  - duplicate statement handling.
- Route tests for status and pendencies.
- Sales -> Finance contract test using persisted store.
