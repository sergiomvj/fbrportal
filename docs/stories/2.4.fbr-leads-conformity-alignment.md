## Story

**As a** outbound growth owner,  
**I want** FBR-Leads reviewed and corrected against the original outbound intelligence concept,  
**so that** lead discovery, validation, scoring, cadencing, and SQL delivery to Click fully match the intended product.

## Acceptance Criteria

1. The implemented Leads scope is reviewed against `fbr-portal-docs/04_LEADS/PRD_SPEC_TASKLIST.md`, `fbr-portal-docs/04_LEADS/TASKS_LEADS.md`, and `fbr-portal-docs/04_LEADS/concept.html`, with all shipped capabilities classified as conformant, partial, or missing.
2. Core leads functionality matches the original concept: capture, email validation, ICP matching, scoring, copy generation, cadence execution, SQL delivery, and operator observability.
3. Information flow matches the original ecosystem design: lead sources feed Leads, enrichment/qualification stays inside the Leads pipeline, and only qualified SQL payloads cross into Click through the documented event contract.
4. Human/agent responsibilities, warming, deliverability, scoring, and sequence behavior match the original docs and general security model.
5. All identified Leads gaps produce concrete code and test changes without adding new outbound features outside the original docs.
6. Existing Leads story `1.5.fbr-leads-outbound-completion.md` is revalidated against the original concept, and any residual drift is explicitly closed.
7. Quality gates pass for the final conformity increment: `npm run lint`, `npm run typecheck`, `npm test`, and `npm run build`.
