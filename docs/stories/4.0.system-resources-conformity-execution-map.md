# Story 4.0: System Resources Conformity Execution Map

## Status

Draft

## Story

**As a** program owner,  
**I want** every remaining PRD gap in system resources split into executable stories,  
**so that** the portal can move from functional slices and in-memory stores to durable, auditable, PRD-compliant runtime architecture.

## Background

The review in `docs/revisao-conformidade-prds-recursos-sistema-2026-05-14.md` concluded that FBR-MKT is conformant in the current state, but the rest of the portal is not yet conformant at the system-resource level.

The main architectural gap is not screen coverage. It is runtime integrity:

- core domain data still stored in process memory;
- fixtures still used as operational state;
- migrations created but not wired to route/runtime code;
- async jobs simulated in local arrays instead of persistent queues/tables;
- audit trails not proven append-only in runtime;
- SSE/realtime often local to a single process;
- global rate limit still process-local outside MKT;
- integration readiness sometimes hidden behind `skipped` responses.

## Non-Negotiable Rules

1. Do not mark any story complete if its core runtime still depends on arrays, fixtures, or process-local `Map` as source of truth.
2. Do not count migrations alone as compliance. Route/runtime code must use them.
3. Do not count tests alone as compliance if they only validate process-local behavior.
4. Every closure must include code, tests, evidence in the relevant 2.x/3.x/4.x story, and green gates.
5. If a PRD requires Redis, BullMQ, n8n, Celery, Supabase Realtime, or storage and the repo uses an equivalent, record the equivalence explicitly in the story evidence.
6. If an integration can return `skipped` because an env var is missing, the story must define whether that is acceptable for local dev only or whether production readiness must fail.

## Global Acceptance Criteria

1. Every system-resource gap from the review has an executable story.
2. Each story states what must be implemented, changed, or refactored.
3. Each story names evidence files, likely code paths, and PRD references.
4. Each story defines tests that prove durable behavior, not only happy-path in-memory behavior.
5. Execution order protects dependencies: Leads foundation before Leads scoring, Click before downstream Sales closure, Finance worker before final finance conformity.
