## Story

**As a** platform owner,  
**I want** the MKT runtime architecture and delivery baseline implemented exactly as required by the source artifacts,  
**so that** every downstream MKT story runs on an approved queue, storage, integration, and security foundation instead of undocumented mocks or implicit local assumptions.

## Acceptance Criteria

1. The current repository has an explicit approved path map for the MKT runtime surfaces required by the source artifacts: proxy/API routes, worker processors, storage adapters, bridge/integration hooks, tests, and UI consumption points.
2. Queue topology exists for the source-documented MKT flow with at minimum `mkt:upload`, `mkt:estrategia`, `mkt:copy`, `mkt:calendario`, `mkt:export`, and retryable downstream delivery behavior for FBR-Click handoff when required by the source docs.
3. Queue configuration follows the documented retry/backoff/retention invariants or an approved repository-accurate equivalent, including observable terminal failure state.
4. Upload and export storage contracts exist for document ingestion and generated outputs, including signed-download or equivalent time-limited access semantics for exported files.
5. Security baseline exists for MKT-sensitive routes and file handling: MIME validation, extension validation, size limits, route-level rate limiting for upload/generation/export/chat paths, and a documented malware-scan hook or equivalent safety checkpoint where the concept requires one.
6. LLM/runtime orchestration contracts are documented and implemented for queue-to-agent execution, health-check visibility, and fallback-aware status reporting to the extent required by the current repository architecture.
7. Notification/delivery baseline exists for strategy-ready, export-ready, and critical-failure events through the approved portal/bridge/email surfaces required by the source artifacts.
