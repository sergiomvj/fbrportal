## Story

**As a** finance manager,
**I want** payables, approvals, cost centers, P&L, and cross-module financial events,
**so that** expenses are governed, allocated, and visible by company and cost center.

## Acceptance Criteria

1. `GET /api/proxy/finance/pagamentos`, `POST /api/proxy/finance/pagamentos`, and `POST /api/proxy/finance/pagamentos/{id}/aprovar` exist and register audit events.
2. Approval workflow enforces no self-approval, no skipping levels, rejection requires observation, already approved returns 409, and recurring payments schedule the next payment through the documented automation boundary.
3. Approval thresholds/role mapping from TASK-FI06 are implemented without silently merging conflicting visual thresholds from TASKS_FINANCE.
4. `GET /api/proxy/finance/centros-custo` returns a tree up to 3 levels and current-month spend from approved payments.
5. `GET /api/proxy/finance/pl/{empresa_id}` returns revenue, expenses, profit, 6-month history, and budget variance.
6. Generic `financial.event` intake creates a pending payable linked to the correct cost center.
7. Frontend includes payables table/approval workflow plus cost center tree and P&L table.

## Dev Notes

### Source Context

- Payables and approval routes are specified in `fbr-portal-docs/03_FINANCE/PRD_SPEC_TASKLIST.md#task-fi06-api-routes-de-pagamentos-e-aprovacao-com-registro-em-audit`.
- Cost center and P&L routes are specified in `fbr-portal-docs/03_FINANCE/PRD_SPEC_TASKLIST.md#task-fi07-api-route-de-centros-de-custo-e-pl-por-empresa`.
- Payables UI is specified in TASK-FI08; cost center/P&L UI is specified in TASK-FI09.
- `financial.event` from any module carries cost type, description, amount, company id, optional cost center, recurrence flag, reference date, and source module. [Source: `fbr-portal-docs/00_GERAL/DOCUMENTO_GERAL.md#117-evento-financialevent-qualquer-modulo-fbr-finance`]

### API and Component Notes

- Cross-module communication must use REST event payloads through module APIs; modules must not write directly to another module database.
- Confirmation flows should use ConfirmDialog semantics for approval/rejection/destructive decisions with loading state.
- Status badges should use documented shared semantics, with module-specific accent awareness.

### File Locations

- Suggested proxy routes: `apps/portal/src/app/api/proxy/finance/pagamentos/**/route.ts`, `apps/portal/src/app/api/proxy/finance/centros-custo/route.ts`, and `apps/portal/src/app/api/proxy/finance/pl/[empresa_id]/route.ts`
- Suggested UI routes: `apps/portal/src/app/finance/pagamentos/page.tsx`, `apps/portal/src/app/finance/centros-custo/page.tsx`, and `apps/portal/src/app/finance/pl/page.tsx`
- Suggested shared helpers: `apps/portal/src/lib/finance/{types,schemas,store,fixtures}.ts`

### Dependencies

- Depends on Story 1.4.2 for payables, cost centers, approval flows, audit log, and company isolation.

### Project Structure Notes

- Governance threshold conflict: visual section says `<=R$500 gestor`, `R$500-5k diretor`, `>5k C-level`, while TASK-FI06 specifies approval logic around `<=R$5.000` and `>R$50.000`. Do not silently merge these; request PO/architect clarification if executable policy is ambiguous.

### Testing

- Required tests: approval roles/levels/self-approval, rejection observation, already-approved conflict, recurring scheduling boundary, P&L calculations, financial event intake, and UI approval dialogs.
- Required final gates: `npm run lint`, `npm run typecheck`, and `npm test`.
