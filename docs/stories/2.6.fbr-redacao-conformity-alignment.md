## Story

**As a** newsroom operations owner,  
**I want** FBR-Redação reviewed and corrected against the original automated newsroom concept,  
**so that** sources, article production, media enrichment, editorial review, publication, and downstream signals fully match the intended product.

## Acceptance Criteria

1. The implemented Redação scope is reviewed against `fbr-portal-docs/06_REDACAO/PRD_SPEC_TASKLIST.md`, `fbr-portal-docs/06_REDACAO/TASKS_REDACAO.md`, and `fbr-portal-docs/06_REDACAO/concept.html`, with all shipped capabilities classified as conformant, partial, or missing.
2. Core newsroom functionality matches the original concept: source ingestion, UGC intake where applicable, drafting, media enrichment, editing, publication, and editorial observability.
3. Information flow matches the original ecosystem design: source collection stays within Redação, editorial stages are traceable, and `article.published` reaches downstream consumers through the documented event contract.
4. Human/agent responsibilities, confidence scoring, publication controls, and audit/security behavior match the original docs and the portal-wide general architecture.
5. All identified Redação gaps produce concrete code and test changes without adding new editorial scope outside the original docs.
6. Existing Redação stories `1.7.1` through `1.7.4` and `1.7` are revalidated against the original concept, and any residual drift is explicitly closed.
7. Quality gates pass for the final conformity increment: `npm run lint`, `npm run typecheck`, `npm test`, and `npm run build`.
