# Story 4.11: Cross-Module Rate Limit, SSE/Realtime, Audit, and Readiness Hardening

## Status

Draft

## Story

**As a** platform owner,  
**I want** cross-module system resources hardened consistently,  
**so that** modules do not each solve rate limiting, realtime, audit, and integration readiness in incompatible or unsafe ways.

## Problem to Fix

MKT now has a persistent fail-closed rate limit, persisted jobs, and SSE fallback from persisted state. Outside MKT, the platform still has process-local rate limit and several modules lack a shared readiness/audit/SSE standard.

Current evidence:

- `apps/portal/src/lib/rate-limit.ts` uses a process-local `Map`.
- `apps/portal/src/lib/mkt/security.ts` uses persistent RPC and fails closed.
- `apps/portal/src/lib/mkt/sse.ts` has in-memory live channels but can bootstrap/poll persisted state.
- `apps/portal/src/lib/mkt/notifications.ts` now persists operational notifications in `mkt_notifications`; it no longer uses a process-local array.
- Sales and MKT bridge code can return `skipped` when integration endpoints are not configured.

## What Must Be Implemented

1. Inventory every use of generic rate limit:
   - find all imports of `apps/portal/src/lib/rate-limit.ts`;
   - classify each use as test/dev/prod;
   - document whether it must be persistent.
2. Create shared persistent rate-limit helper:
   - either generalize MKT RPC pattern;
   - or create module-scoped Supabase/Redis equivalent;
   - define fail-open vs fail-closed policy explicitly, defaulting to fail-closed for sensitive operations.
3. Replace production use of process-local buckets:
   - migrate routes to shared persistent helper;
   - keep process-local helper only for tests/dev if needed;
   - add tests for infra unavailable behavior.
4. Define SSE/realtime standard:
   - live channel can be in-memory only as optimization;
   - bootstrap/latest state must come from persisted data;
   - reconnect behavior must be documented;
   - multi-instance limitation must be explicit until Redis/Supabase Realtime is used.
5. Define audit/event standard:
   - event tables append-only where required;
   - updates/deletes blocked by trigger/policy when PRD requires immutable audit;
   - every cross-module event has correlation id and delivery status.
6. Define integration readiness standard:
   - each required integration exposes configured/missing/degraded state;
   - `skipped` is allowed for local-dev only when documented;
   - conformant/prod mode must fail readiness when required endpoint/secret is absent.
7. Add platform-level docs:
   - create or update a short system-resource standard document;
   - link each module story to the standard.

## What Must Be Changed or Reworked

- Do not let modules invent separate hidden readiness semantics.
- Rework generic `rate-limit.ts` or mark it test-only/dev-only.
- Rework route tests that assume process-local counters are production-acceptable.
- Add readiness checks for MKT Click bridge, Sales Finance bridge, WordPress, Design/Social webhooks, and external storage/render providers.

## Acceptance Criteria

1. All generic rate-limit uses are inventoried.
2. Production-sensitive routes use persistent rate limit or documented fail-closed equivalent.
3. SSE endpoints can bootstrap from persisted state or document why they are local-only.
4. Cross-module events include correlation id, status, and delivery evidence.
5. Required integrations expose readiness state and fail conformant mode when missing.
6. Tests cover rate-limit unavailable, readiness missing, and SSE persisted bootstrap.

## Test Requirements

- Unit tests for shared rate-limit helper.
- Route tests for fail-closed unavailable infra.
- Readiness tests for configured and missing env states.
- SSE tests for persisted bootstrap where endpoints exist.
- Static/import regression test for production imports of process-local rate limiter if feasible.
