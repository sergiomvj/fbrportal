## Story

**As a** newsroom operator,
**I want** Redacao worker queues, retry handling, alert tasks, and n8n workflow boundaries defined behind backend services,
**so that** article production can run continuously with observable failure handling.

## Acceptance Criteria

1. Celery queues exist for Monitor, Jornalista, Midia, Editor, Publisher, and Gestor with the concurrency and retry policies defined in parent story 1.7.
2. Celery config routes tasks by queue and sets `task_acks_late=True`, `worker_prefetch_multiplier=1`, `task_soft_time_limit=300`, and `task_time_limit=600`.
3. Editorial chain is implemented as Jornalista -> Midia -> Editor -> Publisher.
4. Retry handler increments retry count, retries below 3 attempts with 60s/120s/240s delays, marks article `etapa='erro'` at 3+ attempts, and creates a `falha_agente` alert.
5. Dead letter queue `redacao.dead_letter` uses 7-day TTL.
6. n8n workflow exports are versioned only after repository path strategy is approved.
7. Tests cover RSS collection, deduplication, exclusion filters, malformed/large feeds, retry/backoff, dead letter behavior, alert creation, OpenClaw timeout, image placeholder, editor rejection loops, and WordPress 5xx retry boundaries.
8. Repository-level gates pass: `npm run lint`, `npm run typecheck`, and `npm test`.

### Celery Worker Architecture

| Queue | Agent | Concurrency | Retry policy |
|---|---|---:|---|
| `redacao.monitor` | Monitor | 4 | exponential backoff, max 5 retries, base delay 60s |
| `redacao.jornalista` | Jornalista | 2 | exponential backoff, max 3 retries, base delay 120s |
| `redacao.midia` | Midia | 3 | exponential backoff, max 3 retries, base delay 90s |
| `redacao.editor` | Editor | 2 | exponential backoff, max 3 retries, base delay 120s |
| `redacao.publisher` | Publisher | 2 | exponential backoff, max 5 retries, base delay 60s |
| `redacao.gestor` | Gestor | 1 | exponential backoff, max 3 retries, base delay 180s |

### n8n Workflow Requirements

- `01_monitor_coleta_rss.json`: cron every configurable interval, read active RSS sources by city, fetch/parse feeds, deduplicate by `fonte_url`, apply exclusion words, insert new articles as `coletado`, retry HTTP errors 3 times with 60s delay, and create `fonte_indisponivel` after 3+ city feed failures.
- `02_monitor_deduplicacao.json`: source-doc companion workflow for monitor deduplication.
- `03_pipeline_editorial.json`: trigger when article reaches `coletado`; Jornalista creates PT-BR `conteudo_pt`/`titulo`, Midia fetches licensed image, Editor approves/rejects, Publisher posts to WordPress, and each step emits SSE updates.
- `04_retry_falha.json`: receives `{ artigo_id, agente, erro }`, increments retry count, reschedules or marks `erro`, and creates `falha_agente`.
