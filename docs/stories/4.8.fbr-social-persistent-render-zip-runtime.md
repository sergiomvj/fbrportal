# Story 4.8: FBR-Social Persistent Production, Render, ZIP, Quality, and Design Sync Runtime

## Status

Draft

## Story

**As a** social production operator,  
**I want** social templates, jobs, artifacts, quality checks, ZIP packages, and Design brand-kit sync persisted,  
**so that** FBR-Social can produce deliverable packages with traceable quality evidence.

## Problem to Fix

Social has migrations and route tests, but runtime still keeps templates, jobs, artifacts, quality checks, agent slots, and agent events in memory.

## What Must Be Implemented

1. Wire Social runtime to existing migrations:
   - inspect `supabase/migrations/20260508000001_create_social_tables.sql`;
   - confirm coverage for templates, jobs, artifacts, quality checks, agent events, brand kit snapshots, ZIP packages;
   - add migration for missing PRD-required storage metadata.
2. Implement persistent store:
   - recommended file: `apps/portal/src/lib/social/store-db.ts`;
   - keep response contracts stable where valid.
3. Persist catalog/templates:
   - network;
   - format;
   - dimensions;
   - safe zones;
   - template version;
   - required brand kit fields.
4. Persist production jobs:
   - briefing;
   - selected networks/formats;
   - queue position/status;
   - agent/task logs;
   - render outputs.
5. Persist artifacts and storage:
   - one record per generated asset;
   - storage path;
   - dimensions;
   - file size;
   - content type;
   - naming convention.
6. Persist quality checks:
   - dimensions;
   - safe zone;
   - file size;
   - logo/color rules;
   - final checklist;
   - block delivery on failed required checks.
7. Implement ZIP package runtime:
   - generate or simulate only behind explicit test boundary;
   - persist package manifest;
   - persist package storage path;
   - include all approved artifacts and manifest metadata.
8. Implement Design -> Social sync:
   - receive brand kit update;
   - persist brand kit snapshot/version;
   - invalidate or mark affected jobs/templates as needing refresh;
   - expose sync status.

## What Must Be Changed or Reworked

- Replace local arrays in `apps/portal/src/lib/social/store.ts`.
- Rework SSE/agent stream so bootstrap/latest state can come from persisted logs.
- Remove fake ZIP success from production path unless an actual package/storage path exists.
- Update 3.6 stories with real persistence evidence.

## Acceptance Criteria

1. Template/catalog data is persisted and versioned.
2. Job creation persists production state and queue position.
3. Rendered artifacts persist storage metadata.
4. Quality checks persist pass/fail details and can block delivery.
5. ZIP package has persisted manifest and storage path.
6. Design brand-kit sync updates persisted Social state.
7. Agent/SSE status can bootstrap from persisted logs.
8. Tests prove runtime does not depend on local arrays.

## Test Requirements

- Migration tests for Social tables/RLS/triggers.
- Store tests for templates, jobs, artifacts, quality checks, packages.
- ZIP manifest test.
- Design sync webhook test.
- SSE bootstrap test from persisted agent events.
- Regression guard against route imports of in-memory store.
