## Story

**As a** commercial intelligence operator,
**I want** FBR-Leads to capture, enrich, score, cadence, monitor domains, and hand off qualified leads,
**so that** FBR-Click receives reliable SQLs with complete qualification context.

## Acceptance Criteria

1. Visual design implements Leads CSS variables, hero/KPIs, 7-stage funnel, pipeline detail cards, 6 agent teams, warming protocol table, and domain health dashboard.
2. Data model includes complete `HandoffPayload`, warming phase type, domain health thresholds, and scoring model with 12 weighted variables.
3. Components include redesigned dashboard, ICP manager, campaign wizard, agent dashboard, and weekly reports.
4. Arva integration supports 6 team slots: Guardioes, Garimpeiros, Analistas, Redatores, Cadenciadores, Inteligencia.
5. Lead source fields, scoring algorithm, ICP matching, domain health, warming, cadence rules, and handoff flow match `PRD_SPEC_TASKLIST.md`.
6. Handoff to FBR-Click emits/uses the complete payload specified for `lead.qualified`.
7. Tests cover source extraction schemas, scoring, ICP matching, domain thresholds, cadence interruption, handoff payload, and UI flows.

#### Critical Issues

1. Story scope is too large for a ready development story. It combines visual redesign, data contracts, frontend components, Arva integration, DB/API/frontend/agent/handoff work from Sprint 1-4, and test coverage in one Draft. The referenced source doc breaks this into LD-V01-LD-V07, LD-D01, LD-C01-LD-C05, LD-A01, and TASK-LD01 through TASK-LD17, with explicit dependencies. This blocks actionable implementation sequencing.
2. File path guidance conflicts with the current repo architecture. Dev Notes list expected production paths `apps/modules/leads` and `backend/api/leads`, while `docs/architecture/arvore-de-origem.md` defines `apps/portal` as the authenticated portal app and the repo currently has no `backend/` directory. The story must specify whether Leads is implemented inside `apps/portal/src/app`, through a new approved backend workspace, or by another architecture-approved path.
3. Acceptance Criteria and tasks are not self-contained enough for @dev. They require reading `PRD_SPEC_TASKLIST.md`, `TASKS_LEADS.md`, and `DOCUMENTO_GERAL.md` to determine exact schemas, endpoints, thresholds, task dependencies, and payload fields. The story should inline the minimum implementation facts needed or split into smaller stories with focused Dev Notes.
4. The "7-stage funnel" requirement is inconsistent across sources. `TASKS_LEADS.md` asks for 7 stages, but `PRD_SPEC_TASKLIST.md` section "Pipeline de Leads (7 Etapas)" enumerates 8 items including Handoff SQL. The story must resolve which source controls the UI and workflow count.
5. Executor assignment is too coarse for the work type. The story contains schema/DB, API/backend, frontend/UI, agent orchestration, and handoff integration. A single `executor: "@dev"` does not match the dynamic executor guidance for mixed database, UI, infrastructure/agent, and integration work. Split by work type or document an approved exception.
