# Story 4.4: FBR-Sales Persistent Partner, Revenue, and Integration Readiness Runtime

## Status

Draft

## Story

**As a** Sales operator,  
**I want** partners, revenues, media kits, anomalies, and cross-module events persisted with readiness checks,  
**so that** Sales can reliably receive closed deals and forward financial events without silent runtime loss.

## Problem to Fix

Sales has integration slices, but the core domain is still in arrays and `Sales -> Finance` can return `skipped` when `SALES_FINANCE_INTAKE_URL` is missing.

Current evidence:

- `apps/portal/src/lib/sales/store.ts` keeps partners, partner events, ad spaces, revenues, anomalies, media kits, and rate cards in memory.
- Finance forwarding can return `SALES_FINANCE_INTAKE_URL_NOT_CONFIGURED`.

## What Must Be Implemented

1. Create Sales persistence migrations:
   - partners;
   - partner events/audit;
   - ad spaces/products;
   - revenues;
   - media kits;
   - rate cards;
   - anomalies;
   - webhook delivery attempts;
   - integration readiness records if needed.
2. Implement RLS:
   - every table scoped by company;
   - audit/event tables append-only or controlled by triggers;
   - role-sensitive access for administrative/commercial operations.
3. Create persistent Sales store:
   - recommended file: `apps/portal/src/lib/sales/store-db.ts`;
   - preserve API response contracts;
   - map domain enums explicitly.
4. Resolve product-state conflict:
   - review `onboarding` vs `prospect` conflict documented in backlog;
   - choose the PRD-authoritative value or record an explicit product decision;
   - update schemas, tests, UI labels, and docs consistently.
5. Rework Click -> Sales:
   - receive `deal.closed` with HMAC;
   - idempotently create/update partner;
   - persist source deal id, amount, product, account owner, and payload;
   - append partner event/audit row.
6. Rework Sales -> Finance:
   - make missing finance endpoint a readiness failure for compliant environments;
   - allow local-dev `skipped` only if explicitly marked as dev/test behavior;
   - persist delivery attempts, response, retry count, next attempt, and terminal failure;
   - implement retry/backoff or persistent pending-delivery status.
7. Rework media kits:
   - persist media kit jobs;
   - persist generated artifact metadata and storage path;
   - avoid synchronous fake success if artifact does not exist.
8. Rework anomalies and alerts:
   - calculate anomalies from persisted revenue/partner state;
   - persist alert events;
   - dedupe recurring alerts.

## What Must Be Changed or Reworked

- Replace array-backed runtime in `sales/store.ts`.
- Rework tests that currently validate only local arrays.
- Add readiness endpoint or health contract for required integrations.
- Update 3.3 stories if they claim completion without persistence.

## Acceptance Criteria

1. `deal.closed` creates/updates persisted partner and partner event.
2. Partner lifecycle changes are persisted and auditable.
3. Revenue records are persisted and company-scoped.
4. Finance forwarding records delivery status and does not silently disappear.
5. Missing required integration config is visible as readiness failure in conformant environments.
6. Media kit generation creates persisted job/artifact records.
7. Anomalies are derived from persisted data and stored with dedupe.
8. Tests cover HMAC, idempotency, persistence, retry/readiness, and audit.

## Test Requirements

- Migration tests for Sales tables/RLS/audit constraints.
- Store tests for partner lifecycle, revenue creation, media kits, anomalies.
- Webhook tests for `deal.closed`:
  - valid HMAC;
  - invalid HMAC;
  - idempotency by event/deal id;
  - update existing partner;
  - missing required fields.
- Sales -> Finance delivery tests:
  - configured endpoint sends event;
  - unconfigured endpoint reports readiness failure for conformant mode;
  - retry attempt is persisted;
  - terminal failure is observable.
