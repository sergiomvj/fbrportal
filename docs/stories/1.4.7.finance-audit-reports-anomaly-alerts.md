### API Contract Notes

Use the source-backed contracts below. Do not add endpoint paths, status values, or payload fields beyond these until an architecture/product decision records them.

#### Audit List

- Endpoint: `GET /api/proxy/finance/audit`.
- Required headers: `X-User-Id` and `X-Module-Source: finance`.
- Query filters: optional `empresa_id`, `entidade`, `acao`, `actor_type`, `actor_id`, `data_inicio`, `data_fim`, `page`, and `limit`.
- `entidade` values from TASK-FI15: `recebimento`, `pagamento`, and `conciliacao`.
- `acao` values from TASK-FI15: `created`, `updated`, `conciliado`, `aprovado`, and `rejeitado`.
- `actor_type` values from TASK-FI15: `human` and `agent`.
- Pagination defaults: `page = 1`, `limit = 50`, maximum `limit = 200`.
- Response shape: `{ success: true, data: AuditEntry[], meta: { page, limit, total, total_pages } }`.
- Audit entry fields: `id`, `empresa_id`, `entidade`, `entidade_id`, `actor_type`, `actor_id`, `actor_nome`, `acao`, `valor_antes`, `valor_depois`, and `created_at`.
- Access: Gestor and Financeiro see audit from their company; Auditor can see all companies; Read-only cannot access audit.

#### DRE Report

- Endpoint: `GET /api/proxy/finance/relatorios/dre`.
- Required headers: `X-User-Id` and `X-Module-Source: finance`.
- Query fields from TASK-FI15: `empresa_id`, `ano`, and optional `trim`.
- Response shape: `{ success: true, data: { empresa_id, empresa_nome, periodo, dre, dmpl } }`.
- DRE calculations must follow TASK-FI15: gross revenue from receivables in period, default 15% taxes when company config is absent, service costs from service/person payments, operational expenses grouped by category, EBITDA as gross profit minus operational expenses, 5% monthly depreciation default, and IR at 15% up to R$20k/month or 25% above.
- Edge case: company with no data returns zeros instead of omitting the DRE shape.

#### Async Export

- Creation endpoint from TASK-FI15: `POST /api/proxy/finance/relatorios/export`.
- Required headers: `Content-Type: application/json`, `X-User-Id`, and `X-Module-Source: finance`.
- Request body fields: `tipo`, `formato`, `empresa_id`, `ano`, optional `trim`, optional `mes`, `inclui_dmpl`, optional `data_inicio`, and optional `data_fim`.
- Allowed `tipo` values: `dre`, `fluxo_caixa`, `conciliacao`, and `audit`.
- Allowed `formato` values: `csv` and `pdf`.
- Validation constraints from TASK-FI15: `ano` between 2020 and 2030, `trim` between 1 and 4 when present, `mes` between 1 and 12 when present, invalid period returns 422.
- Creation response: HTTP 202 with `{ success: true, data: { job_id, status: "processing", estimated_seconds, formato } }`.
- Persisted export status values from TASK-FI15: `processing`, `ready`, and `error`.
- Ready download payload from TASK-FI15: `{ success: true, data: { download_url, expires_at, tamanho_bytes } }`.
- Error payload must expose the persisted `erro_msg` through the repo standard error envelope once that envelope is available from upstream API foundations.
- Expiration state uses persisted `expires_at`; expired downloads must show "Link expirado, gere novamente" in the reports UI.
- The exact status/history/download endpoint paths and response envelopes are not specified in current source artifacts. They require the decision below before implementation.

#### Alert List and Resolve

- Source artifacts require alert routes, alert list, resolve action, unresolved badge, and persisted alerts, but do not define exact endpoint paths or response envelopes.
- Alert list item fields from TASK-FI18 UI contract: `id`, `tipo`, `severidade`, `titulo`, `descricao`, `entidade`, `entidadeId`, `detectedAt`, `resolvedAt`, and `resolvedBy`.
- Allowed `tipo` values: `divergencia_valor`, `atraso_pagamento`, `orcamento_excedido`, and `transacao_suspeita`.
- Allowed `severidade` values: `info`, `warning`, and `critical`.
- Resolve behavior from TASK-FI18: "Resolver" marks an alert as resolved; an already resolved alert must not reopen.
- The exact alert list path, resolve path/method, unresolved-count badge path, and success/error envelopes require the decision below before implementation.

### Anomaly Rules

Use only these source-backed anomaly rules from TASK-FI18 unless PO/architect records a narrower or expanded contract:

- `divergencia_valor`: difference greater than 10% between statement and ledger entry; severity `warning`.
- `atraso_pagamento`: pending payment with `data_vencimento < today - 7 days`; severity `critical`.
- `orcamento_excedido`: `gasto_atual > orcamento_mensal * 1.1`; severity `warning`.
- `transacao_suspeita`: value greater than 3x the average from the last 6 months; severity `critical`.
- Pending reconciliation older than 48h without resolution; severity `warning`.
- Current-month revenue below 80% of the last 3 months average; severity `critical`.
- Critical alerts send Portal Bridge notification with `postMessage({ type: 'NOTIFICATION', payload: { title, body, level } })`.
- Notification rate limit: maximum 5 notifications per minute when many anomalies are detected.
- If Portal Bridge notification fails, the alert must still be saved.
- Duplicate detection must not create duplicate alerts for the same anomaly; source detail suggests uniqueness by `empresa_id`, `tipo`, and `entidade_id`.
