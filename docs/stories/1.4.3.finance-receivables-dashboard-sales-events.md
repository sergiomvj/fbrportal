## Story

**As a** finance manager,
**I want** receivables APIs, dashboard metrics, and Sales payment intake,
**so that** incoming revenue is visible, filterable, and automatically prepared for reconciliation.

## Acceptance Criteria

1. `GET /api/proxy/finance/recebimentos` supports filters for parceiro, status, date range, company, pagination, and company isolation.
2. `POST /api/proxy/finance/recebimentos` validates required fields, positive monetary values, valid company, and returns descriptive 400/422 errors.
3. `PATCH /api/proxy/finance/recebimentos/{id}/conciliar` marks divergence <= 5% as received, divergence > 5% as divergent, and already reconciled items return 409.
4. Dashboard API returns `receita_total`, `a_receber`, `atrasados`, `divergencias`, `projecao_30d`, previous-period delta, revenue by partner, and revenue by company.
5. Sales `payment.received` intake creates pending receivables and can trigger automatic reconciliation.
6. Frontend includes dashboard KPI cards and a receivables table with filters, create modal, validation, badges, sorting, pagination, loading, retry, and empty states.

### Source Context

### API and Component Notes

- Required proxy convention is `/api/proxy/{modulo}/{entidade}[/{id}[/{acao}]]`, with `X-User-Id` injected by the Next proxy and `X-Module-Source` identifying module origin. [Source: `fbr-portal-docs/00_GERAL/DOCUMENTO_GERAL.md#12-padroes-de-api-routes`]
- `recebimentos` status values in the SPEC are `pendente`, `recebido`, `atrasado`, and `divergente`.
- Dashboard KPIs use DataCard-like semantics with icon, label, value, delta, trend, loading, error/retry, and finance accent color.
- Cross-module events are REST `POST` requests with typed JSON payload plus `X-User-Id` and `X-Module-Source`; modules must not write directly to another module database. [Source: `fbr-portal-docs/00_GERAL/DOCUMENTO_GERAL.md#11-contratos-cross-module-payloads-detalhados`]

### Sales Payment Intake Contract Scope

The source artifacts currently define only this traceable contract:

- Event name: `payment.received`.
- Direction: FBR-Sales to FBR-Finance.
- Transport: REST `POST`, typed JSON payload, `X-User-Id`, and `X-Module-Source`.
- Payload envelope: `{ event: 'payment.received', data: { ... } }`.
- Required data fields by TypeScript interface: `parceiro_id`, `parceiro_nome`, `empresa_id`, `valor`, `moeda`, `periodo_ref`, `data_recebimento`, and `tipo_parceria`.
- Optional data fields by TypeScript interface: `comprovante_path` and `notas`.
- Finance action: create a receivable with status `pendente` and attempt automatic reconciliation.

Do not implement an intake route name, response shape beyond the standard success/error envelopes, idempotency key, duplicate/conflict behavior, or automatic reconciliation eligibility rules until the decision below is recorded.

**Decision needed:** define the executable Sales `payment.received` intake contract for this repo.
