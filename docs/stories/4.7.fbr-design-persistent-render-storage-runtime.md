# Story 4.7: FBR-Design Persistent Brand Kit, Render Job, Storage, and Integration Runtime

## Status

Draft

## Story

**As a** design operator,  
**I want** clients, brand kits, assets, render jobs, variants, reviews, gallery items, and integrations persisted,  
**so that** FBR-Design can produce real reusable creative assets for Sales, Social, and MKT.

## Problem to Fix

Design has migrations, but runtime still keeps brand kits, jobs, templates, agent slots, and activity logs in memory.

## What Must Be Implemented

1. Wire Design runtime to existing migrations:
   - inspect `supabase/migrations/20260508000001_create_design_tables.sql`;
   - confirm required tables for clients, brand kits, jobs, variants, templates, assets, reviews, activity log;
   - add missing columns/tables with migration if PRD-required.
2. Implement persistent store:
   - recommended file: `apps/portal/src/lib/design/store-db.ts`;
   - map existing API contracts to DB rows.
3. Persist clients and brand kits:
   - client/company ownership;
   - logo/color/font/token metadata;
   - storage path for uploaded logos/assets;
   - versioning or updated timestamp.
4. Persist render jobs:
   - briefing;
   - status;
   - queue position;
   - selected template;
   - variants;
   - review findings;
   - output artifact metadata and storage path.
5. Implement storage contract:
   - no fake artifact success without a persisted storage path;
   - support signed URL or proxy download according to existing repo pattern;
   - record upload/render failures.
6. Implement review/approval:
   - persist auto-review checks;
   - persist human approval/rejection;
   - append activity/audit events.
7. Implement integrations:
   - Design -> Sales approval/use of media kit creative;
   - Design -> Social brand kit update webhook;
   - Design -> MKT export/asset use when applicable;
   - persist delivery attempts and readiness.

## What Must Be Changed or Reworked

- Replace in-memory `brandKits`, `jobs`, `templates`, `agentSlots`, and `activityLog` as runtime source.
- Rework tests that only validate local state.
- Ensure existing migrations are not just tested but actually used by routes.
- Update 3.5 stories if they claim completion without persistent runtime.

## Acceptance Criteria

1. Brand kit CRUD persists data and asset storage metadata.
2. Render job creation persists briefing, queue/status, and variants.
3. Review/approval updates persisted job/variant state and activity log.
4. Gallery reads persisted artifacts with storage metadata.
5. Integrations to Sales/Social/MKT persist delivery attempts or explicit readiness failures.
6. Tests prove route/runtime code uses persistent store.

## Test Requirements

- Migration tests for Design tables/RLS/triggers.
- Store tests for brand kits, jobs, variants, review, gallery, templates.
- Storage tests with mocked storage client.
- Webhook/integration tests for Social/Sales/MKT delivery attempts.
- Regression guard against route imports of in-memory store.
