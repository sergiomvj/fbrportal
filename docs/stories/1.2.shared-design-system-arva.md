## Story

**As a** FBR Portal module builder,
**I want** shared visual primitives and a reusable Arva agent integration contract,
**so that** every module can reuse the same design language and agent-linking behavior without duplicating CSS tokens, UI patterns, or Arva API client logic.

## Acceptance Criteria

1. Shared visual tokens are exported from `packages/ui` and consumed by the portal without duplicating module CSS: fonts `Outfit`, `Inter`, `JetBrains Mono`; dark palette variables; brand orange CTA token; module accent variables; grid background; glow effects; glass surface pattern; card pattern; KPI grid; hero pattern; and reduced-motion-safe animations. [Source: `fbr-portal-docs/00_GERAL/DOCUMENTO_GERAL.md#4-padroes-visuais-design_standards`, `fbr-portal-docs/TASKLIST_GERAL.md`]
2. `AgentPicker` is exported from `packages/ui` and renders an "Incluir Agente" entry point, modal grid, tag filters, empty/error/loading states, linked-agent cards, heartbeat indicator for active agents, and "AGENTE" badge. It displays Arva agent avatar, name, role, tags, and status. [Source: `fbr-portal-docs/TASKLIST_GERAL.md#glb-10-agent-card-pattern-via-arva-platform`]
3. `packages/arva-integration` exists with typed exports `listAgents(companyId)`, `resolveAgent(fbrchatId)`, and `openChat(agentId, userId)`, plus `ArvaAgent`, `AgentIdentity`, `AgentPersona`, `AgentRuntime`, and `AgentPerformance` types. [Source: `fbr-portal-docs/TASKLIST_GERAL.md#glb-11-arva-integration-package`]
4. Arva API calls use the documented endpoints only: `GET /api/agents?company_id={companyId}`, `POST /api/integrations/fbrchat/resolve-agent`, and `POST /api/integrations/arva/chat/open`. Request/response handling covers success, empty result, non-2xx response, malformed payload, and network failure.
5. `ARVA_BASE_URL` and `ARVA_FBRCHAT_SHARED_TOKEN` are documented in `.env.example` without exposing real values. `ARVA_FBRCHAT_SHARED_TOKEN` must never be required as `NEXT_PUBLIC_*`, must not be imported into client components, and Arva calls that need the shared token must cross a server-side boundary.
6. Module-facing integration is explicit: modules consume a shared CSS import/export surface from `packages/ui`, mount the same "Incluir Agente" action, pass module tag filters, receive selected linked agents through callbacks, and render linked agents with heartbeat state. No module duplicates Arva client logic. [Source: `fbr-portal-docs/TASKLIST_GERAL.md#glb-12-agent-panel-em-cada-modulo`]
7. This story must not implement module-specific business workflows for Click, Finance, Leads, Marketing, Redacao, Sales, Social, VideoFlow, or Design. It only provides reusable primitives, contracts, and tests proving workflow-specific code is absent from this story.
8. Quality gates pass: lint, typecheck, tests, and accessibility coverage for modal focus, keyboard navigation, labels, status semantics, visible focus, reduced motion, and color contrast.

### API Contract

The story must implement only the Arva endpoints already listed in `TASKLIST_GERAL.md`:

- `GET /api/agents?company_id={companyId}` for `listAgents(companyId)`.
- `POST /api/integrations/fbrchat/resolve-agent` for `resolveAgent(fbrchatId)`.
- `POST /api/integrations/arva/chat/open` for `openChat(agentId, userId)`.

Expected client behavior:

- Return typed success data when response shape is valid.
- Return or throw normalized errors for non-2xx, malformed payload, missing env, and network failures.
- Never expose `ARVA_FBRCHAT_SHARED_TOKEN` through public env vars, browser bundles, logs, or rendered props.

### Module Consumption Contract

Modules should consume:

- Shared tokens/styles from `packages/ui`.
- `AgentPicker` props for `companyId`, optional `moduleId`, module tag filters, linked agents, and selection callback.
- Arva functions through shared package/server boundary rather than per-module client copies.

Modules must not consume:

- Raw `ARVA_FBRCHAT_SHARED_TOKEN`.
- Duplicated CSS token files.
- Module-specific workflow code from this story.

### File Locations

- Expected shared UI path: `packages/ui`.
- Expected Arva client path: `packages/arva-integration`.
- Expected app-level style integration path should align with the Story 1.1 portal app structure under `apps/portal`.
- Expected environment documentation: root `.env.example` and/or `apps/portal/.env.example`, aligned with existing env convention.

### Testing

- Unit-test token exports and Arva client functions.
- Component-test `AgentPicker` loading, success, empty, error, tag filtering, selection, linked-agent cards, and status display.
- Accessibility-test modal focus trap, keyboard navigation, visible labels, reduced motion, and contrast.
- Add a scope guard proving module-specific workflows remain out of this story.
