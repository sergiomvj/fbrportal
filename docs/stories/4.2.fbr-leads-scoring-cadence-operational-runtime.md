# Story 4.2: FBR-Leads Scoring, Cadence, Deliverability, and Click Handoff on Persistent Runtime

## Status

Draft - depends on Story 4.1

## Story

**As a** sales development lead,  
**I want** captured leads scored, matched to ICPs, progressed through cadence, and handed off to Click using persisted state,  
**so that** lead qualification is explainable, auditable, and recoverable.

## Problem to Fix

Story 3.1.2 is partial: ICP CRUD and seven stages exist, but score is fixed in fixtures. Story 3.1.3 and 3.1.4 still need persistent cadence, domain health, agent logs, reports, and end-to-end handoff.

This story must only start after Story 4.1 replaces source capture runtime with Supabase.

## What Must Be Implemented

1. Implement the PRD scoring engine:
   - map the 12 scoring variables from `fbr-portal-docs/04_LEADS/PRD_SPEC_TASKLIST.md`;
   - represent each variable as a named rule with weight, input source, positive/negative contribution, and explanation;
   - implement bonus and penalty rules;
   - persist the final score and the rule-by-rule breakdown;
   - never calculate score only in UI.
2. Implement ICP matching against persisted ICP definitions:
   - compare segment, company size, geography, budget, fit signals, and exclusions present in the PRD;
   - persist `matched_icp_id`, confidence, and explanation;
   - handle no-match explicitly.
3. Implement official seven-stage lifecycle:
   - store stage changes in persisted history;
   - reject invalid transitions;
   - record actor, previous stage, next stage, reason, and timestamp;
   - keep dashboard/funnel derived from persisted state.
4. Implement cadence runtime:
   - persist cadence plans and touch status;
   - track the four PRD touches;
   - store email template rendered payload or rendering inputs;
   - pause cadence on bounce, opt-out, invalid email, positive reply, or domain health violation;
   - expose status in route/API.
5. Implement domain deliverability:
   - persist domain health metrics;
   - track bounce rate, warmup state, daily limit, and pause reason;
   - enforce domain limits before scheduling/sending a touch;
   - add readiness status for external providers if the repo has no real mail integration yet.
6. Implement agent and report evidence:
   - persist agent task/log records for captacao, scoring, cadencia, and monitoramento;
   - generate reports from persisted data, not fixtures;
   - expose enough observability to diagnose why a lead did or did not reach Click.
7. Revalidate Click handoff:
   - handoff only after score/ICP/cadence rules qualify the lead;
   - send `lead.qualified` with score breakdown and ICP match;
   - persist handoff result and retry/error state;
   - keep existing Click contract tests and add end-to-end Leads -> Click contract.

## What Must Be Changed or Reworked

- Remove fixed scores from fixtures as runtime source of truth.
- Rework dashboard KPIs so they query persisted leads and stages.
- Rework cadence/domain routes that depend on local arrays.
- Replace any "always healthy" or "always qualified" hardcode with explicit readiness, blocker, or persisted status.
- Keep demo fixtures only as test seeds.

## Acceptance Criteria

1. A lead captured through Story 4.1 receives a persisted, explainable score.
2. ICP matching is persisted and visible through API/UI data contracts.
3. Seven official stages are enforced through validated persisted transitions.
4. Cadence touches are persisted with status, timestamp, template, and failure reason.
5. Domain health can pause cadence and is visible to operators.
6. Reports and dashboard metrics derive from persisted leads/cadences/handoffs.
7. `lead.qualified` includes score and ICP context and is sent to Click only when rules pass.
8. Tests cover scoring, ICP, stages, cadence pause, domain limit, and Click handoff.

## Test Requirements

- Unit tests for each scoring variable and bonus/penalty rule.
- Unit tests for ICP match and no-match.
- Store/route tests for persisted score and stage history.
- Cadence tests for:
  - four touches;
  - bounce pause;
  - opt-out pause;
  - domain daily limit;
  - positive reply handoff.
- Contract test for Leads -> Click `lead.qualified` including score breakdown.
- Dashboard aggregation test using persisted store mocks.
