# Story 4.6: FBR-Redacao Persistent Editorial Pipeline, Workers, WordPress, and Alerts

## Status

Draft

## Story

**As a** newsroom operator,  
**I want** RSS, UGC, articles, editorial stages, worker logs, WordPress publishing, and alerts persisted,  
**so that** Redacao becomes a real editorial system rather than a mock surface.

## Problem to Fix

Redacao still stores articles, RSS sources, UGC submissions, alerts, and agents in process-local arrays. The PRD requires Supabase, RLS, full-text search, workers, n8n/Celery-style orchestration, WordPress integration, media storage, SSE/logs, and alerts.

## What Must Be Implemented

1. Create or complete Redacao migrations:
   - articles;
   - RSS sources;
   - UGC submissions;
   - editorial stages/history;
   - agent tasks/logs;
   - media assets;
   - WordPress publication records;
   - alerts;
   - audit events.
2. Enable RLS and indexes:
   - company/user isolation;
   - full-text search fields/indexes for Portuguese content;
   - append-only audit/event history where required.
3. Create persistent Redacao store:
   - recommended file: `apps/portal/src/lib/redacao/store-db.ts`;
   - preserve route contracts unless PRD requires correction.
4. Implement RSS intake:
   - persist source definitions;
   - run fetch/import job or documented equivalent;
   - dedupe by URL/hash/title similarity;
   - create article draft candidates.
5. Implement UGC intake:
   - persist submissions;
   - validate required fields;
   - moderate status;
   - convert approved UGC into article/pauta pipeline.
6. Implement editorial pipeline:
   - Monitor/Coletor;
   - Jornalista;
   - Midia;
   - Editor;
   - Publisher;
   - persist transitions, actor, timestamp, and reason.
7. Implement WordPress publishing:
   - route/service for WordPress REST API;
   - persist external post id/status;
   - handle media upload path;
   - store failure and retry state.
8. Implement observability:
   - persisted agent logs;
   - SSE or polling endpoint backed by persisted latest logs;
   - critical alerts persisted and visible in portal notifications if applicable.

## What Must Be Changed or Reworked

- Replace `apps/portal/src/lib/redacao/store.ts` arrays as runtime source.
- Remove mock-only pipeline assumptions from route tests.
- Add explicit integration readiness for WordPress/n8n if real external services are not configured.
- Update Redacao 2.x/3.x stories to reflect actual persistent closure.

## Acceptance Criteria

1. RSS source CRUD persists to DB.
2. RSS import creates persisted article candidates and dedupes safely.
3. UGC submissions persist and can enter editorial pipeline after moderation.
4. Articles move through persisted editorial stages.
5. WordPress publish attempts persist external status, id, and errors.
6. Agent logs and alerts are persisted and observable.
7. Search uses persisted article/content data.
8. Tests prove runtime does not depend on local arrays.

## Test Requirements

- Migration tests for Redacao schema/RLS/indexes.
- Store tests for RSS, UGC, articles, stage transitions, alerts.
- Worker/service tests for RSS import and WordPress publish with mocked external responses.
- Route tests for intake, pipeline movement, publish, alerts, and logs.
- Regression guard against route imports of in-memory store.

- Redacao core runtime is persistent and observable.
- WordPress integration has either real configured delivery or explicit readiness failure.
- Gates pass and evidence is updated.
