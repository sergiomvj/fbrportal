## Story

**As a** module owner,  
**I want** the FBR-MKT agent layer and operational surfaces to match the concept,  
**so that** Extrator, Estrategista, Redator, Calendário, Exportador, and Onboarding run in observable, governable, and source-accurate order instead of appearing as decorative labels.

## Acceptance Criteria

1. The module exposes 6 MKT agent slots through the approved Arva/OpenClaw integration surface: Extrator, Estrategista, Redator, Calendário, Exportador, and Onboarding.
2. Agent activity is observable through persistent action logs, queue/job status, and UI surfaces that reflect real execution state.
3. Operational notifications defined by the concept are represented through the approved portal/bridge surfaces for uploads completed, strategies generated, exports ready, and critical failures, including the dedicated MKT communication path where applicable.
4. The runtime exposes health/observability signals for queue and LLM-related MKT operations, including queue state, failure state, and fallback-aware execution reporting to the extent required by the current repository's approved architecture.
5. The agent/runtime layer reflects the source-documented operational chain for queue-triggered agent execution rather than isolated decorative cards.
6. Human-approval requirements for irreversible actions remain explicit and enforceable through the agent/action model.
7. The current static agent page and labels are replaced or upgraded so they no longer imply nonexistent execution capability.
